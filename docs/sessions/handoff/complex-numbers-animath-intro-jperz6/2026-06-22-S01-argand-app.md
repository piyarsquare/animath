# Argand — entry-point app for complex numbers (affine map → quadratics)

## Summary

This branch is a **new app, `Argand`** (`#/argand`), built around the affine map
**`f(z) = α₁·z + α₀`** — the complex cousin of `y = mx + b` — and grown over the
session into a small but rich explorer: three feeds (Point / Shape / Grid), three
number systems (complex / dual / split via a `p = j²` slider), iteration & fixed
points, a "view from z*" recenter, and a **quadratic** mode (`α₂z² + α₁z + α₀`) whose
Point feed shows `f(z)` as the tip-to-tail sum of its terms. It is **immersive**
(fill-the-screen plot) with a bottom control HUD. Build is green, lint clean.
`origin/main` is merged in; the card sits **just after Complex Particles** in the
gallery. Ready to push to main. The stated next step is **consolidation** — the
feature accumulation has left `ArgandPlane.tsx` branchy.

## What changed

The app went through a major **reframe** mid-session: the original "drag two numbers
a, b; combine with ×/+ mode" model (with Curve / Plane / Repeat / Mean views) was
**retired** in favor of one function `f(z) = α₁z + α₀` applied to a Point, a Shape, or
the whole Grid. Net result was ~200 lines *lighter* even as features were added.

Capabilities now:

- **Coefficients as colored, lockable handles** (`z` cyan, `α₁` orange ◆, `α₀` violet ▢,
  `α₂` pink ▲, `f(z)` emerald, `z*` gold), color-matched to the on-screen equation.
- **Two honest legs** (×α₁ spiral, +α₀ slide) closed into a **loop** with an all-at-once
  diagonal return; arc-length-paced Play with a min-sweep floor; labeled stops.
- **Fixed point** `z* = α₀/(1−α₁)`, **View from z\*** (recenter → pure spiral), and
  **Iterate** (orbit `z→f(z)→f²(z)…`).
- **System `p = j²`** (persistent bottom pill + slider with its own Play that morphs
  complex↔split): rotation / shear / boost, with the `x²−p·y²=1` unit curve + null cone.
- **Quadratics** (Degree pill): grid/shape **bend** (sampled), **two fixed points** + a
  **critical point**, literal `fᵏ(z)` orbits, and the **term-sum** Point animation.
- **Grid**: Cartesian | Polar, a Grid-size slider, optional **domain coloring**.
- **Immersive** plot, **bottom control HUD** (feed switcher + t & j² scrub/play), and an
  **on-screen equation** overlay — all surviving fullscreen.

The math lives in `complexOps.ts` (pure, with throwaway-verified helpers); all rendering
in `ArgandPlane.tsx`; state + panels + HUD in `Argand.tsx`.

## Open / not done

- **Consolidation (the headline next step).** `ArgandPlane.tsx` (~600 lines) has grown
  many `degree===1 / degree===2`, `feed`, and `gridColor` render branches. A pass to
  unify the feed renderers over the shared map/path engine, and to factor the grid
  drawing, would pay down debt before the next feature.
- **Degree-2 Play wraps rather than loops.** The affine "diagonal return" was special to
  the line; the quadratic term-sum loop and the shape/grid ramp use a triangle/wrap and
  have a small replay jump. The static pictures are always complete, so it's cosmetic —
  but a true closed loop would be nicer.
- **Fixed points/critical point are exact only in the complex system.** In dual/split,
  the quadratic formula leans on a system square root that can fall back in degenerate
  regions, so those markers are approximate there.
- **Domain coloring is per-segment** → many SVG lines on a dense grid; can feel heavy
  while dragging on a big screen. A per-line gradient would be lighter.
- **Real-device pass wanted** (`phone-needed`): fullscreen-flush HUD and the phone HUD
  were verified by reasoning + headless screenshots, not on a physical phone.

## Context

- **Verification is headless**: `scripts/shoot.mjs '#/argand' out.png` (+ `SEED_LS` to
  seed persisted state, `VIEWPORT=390x844` for phone). Math helpers were checked with
  one-off `npx tsx` scripts against direct evaluation / `f(z*)=z*`.
- **`p` semantics**: only the *sign* of `p = j²` matters (complex / dual / split); the
  degenerate domains where `powRealG` can't form a continuous power are why iteration
  paths are drawn as straight segments between true iterates there (this was the "paths
  don't connect" bug Dan caught).
- The earlier **generalized-systems + Repeat/Mean** views were built and then retired in
  the reframe — don't resurrect Means (Dan found it unhelpful); Repeat's idea ("×b as
  repeated addition") could return as a Point-feed toggle if wanted.
- Append-only shared-file discipline held through the `main` merge; the only conflict was
  the SolidWorlds line in CLAUDE.md (resolved keeping both apps' lines).

## Self-reflection

1. **What would you do with another session?** The consolidation pass on
   `ArgandPlane.tsx`: collapse the feed/degree/grid render branches onto the shared
   map+path engine, and give the degree-2 Play a genuine closed loop. Then optionally a
   cubic/general-degree step (the coefficient array already supports it) and a real-phone
   pass.
2. **What would you change about what you produced?** I twice replaced an animation I had
   just shipped (curve "collapse" loop → arcs; Horner chain → term-sum). Good responsiveness,
   but I could have asked the disambiguating question *before* building the first version
   each time and saved a round.
3. **What were you not asked that you think is important?** Whether Argand should
   eventually **supersede Plane Transform** outright (the CLAUDE.md line says
   "successor-in-progress") — that's a catalog decision worth making explicitly before
   both live side by side indefinitely.
4. **What did we both overlook?** No unit tests were added for the new pure math
   (`complexOps.ts`) despite it being eminently testable and the repo having a vitest
   harness; I verified with throwaway scripts that weren't committed. A small
   `__tests__/complexOps.test.ts` would lock in the affine/poly/fixed-point invariants.
5. **What did you find difficult?** Keeping the animation *semantics* coherent across the
   degree-1 loop, the degree-2 term-sum, iteration, and the three feeds — each combination
   wants a slightly different "what's moving," and the lut/stops/mover matrix got branchy.
6. **What would have made this task easier?** A live device in the loop for the phone /
   fullscreen interaction work, and committing the math-check scripts as real tests from
   the start.
7. **Follow-up value:** LOW — the app builds, lints, and is verified feature-by-feature;
   follow-up is consolidation and polish, not correctness. (Bump to MEDIUM only for the
   dual/split quadratic fixed-points, which are approximate.)
