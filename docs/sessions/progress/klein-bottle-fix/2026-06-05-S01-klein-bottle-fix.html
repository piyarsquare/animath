  7. 02:30

    Milestone

    Footprint model rebuilt: one print per step, on the side you walk

    Why: User spec — "the character should emit one F per move and on only one side of the surface, the one he is walking on. If he walks the twisted direction he reaches the start and finds his footsteps on the other side of the world (through the floor) and reversed (but pointing the same forward direction)… the new F is mirror opposite the one under the ground. Crossing the roll direction this does not happen." The old model was wrong: it stored the trail in a fixed base frame and drew it in every cell on both sides (a permanent top + reflected-under pair), tied to absolute cell parity.

    • The trail is now stored in true cover coordinates (one append per step, on the player's side) and decoupled from the cell groups — its own set of meshes sharing the one trail buffer.
    • Each frame the trail is tiled relative to the player's own cell: the home copy is the identity (your own prints always upright, on top), even-Δcolumn neighbours are translations (stay on top), and odd-Δcolumn neighbours — which differ from you by one twist, an orientation flip — get scale(1,−1,−1): reflected under the glass and z-mirrored, so the prints read reversed with the same forward heading (the twist mirrors the transverse axis, not your heading).
    • Consequence (verified by hand on the z≈0 core): walk +x one cell, your prior cell's prints reappear directly under the new ones, reversed; two cells later they're back on top, upright. Roll (Δcolumn even, z-translation) never flips side. This is exactly the spec.
    • Removed the per-cell fp/ufp meshes; kept the underside decor. npm run build: passed.

  15. 22:40

    Finding

    Diagnosed the current Klein scheme + the design fork

    Why: The user reports walking the Klein bottle and ending up "on the other side of the floor / up is now down," and wants the flip to feel smooth like the Möbius walk — but the current flat engine can't produce that experience, so I needed to pin down what it actually does before proposing a fix.

    Read flatEngine.ts, corridorEngine.ts, engine.ts, TopologyWalk.tsx, EXPLAINER. Key facts:

    • The Klein bottle exists only in the flat family (SURFACES in engine.ts). The corridor family has loop / Möbius / double / triple / trefoil — no Klein.
    • The flat engine is intrinsically flat: camera up is hard-pinned to (0,1,0) and eye height to +EYE every frame (flatEngine.ts:205,211). The "flip" is realised only by z-mirroring odd columns of the tiled universal cover (sz = klein && (I&1) ? -1 : 1, line 223). The player never rolls or flips; you only see landmarks/footprints come back mirror-reversed.
    • So the "up is now down" experience the user describes is literally not producible by the current flat Klein — that sensation belongs to the Möbius corridor, whose smooth roll comes from riding a continuously-twisting swept frame (up = f.b, the binormal from frameAt(); corridorEngine.ts:281,304,306).
    • Mathematically the current flat scheme is a correct flat Klein bottle (glide-reflection gluing). So "not correct" is a presentation disagreement, not a math bug — the user wants to feel the orientation flip, which the intrinsic-flat design deliberately suppresses.

    Crucial constraint to surface: the Klein bottle cannot embed in ℝ³ without self-intersection, so there is no "free" globally smooth walk-on tube the way the Möbius strip has one. A smooth roll is possible only via an immersion (figure-8 / pinched bottle). This makes the next step a genuine design fork → asking the user.
