How a complex number becomes a color

Domain coloring in the animath viewers — turning a value w = u + iv into a pixel you can read

Every picture in the complex viewers is painted by a single rule that turns a complex number into a color. Once you know the rule, the colors stop being decoration and become data: you can read a function's zeros, poles, and growth straight off the image. This guide is that rule — the calcColor routine that runs once per particle — and the ways you can re-aim it. It is the companion to What the functions do, which used the coloring to read functions; here we open up the coloring itself.

Code is quoted from the viewers' shader (src/animations/ComplexParticles/shaders/index.ts, the calcColor function) and the shared palette library (src/lib/colormaps.ts). The applets are live; drag to orbit.

1The wheel: phase to hue, magnitude to brightness

Write the value in polar form, w = r·eiθ. The default coloring gives each of its two parts one job: the angle θ picks a hue by walking once around the color wheel, and the magnitude r sets the brightness, rising from dark near zero to full far out:

hue = θ / 2π  (mod 1)  ·  brightness = ½(1 + tanh(ln r))

shaders/index.ts : 336–352 (calcColor, abridged)

float r = length(w);
float angle = atan(w.y, w.x);
// phase (default quantity): the angle, mapped once around the wheel
float param = angle / TAU + 1.0;
float hue = fract(param + hueShift + uBranchHue);
// magnitude (default brightness): log-squashed so it never clips
float val = 0.5 * (1.0 + tanh(log(r + 1e-6)));
...
return hsv2rgb(vec3(hue, saturation, val)) * intensity;

The tanh(log r) squash is what keeps the brightness usable: a function's magnitude ranges over many orders of magnitude, and a raw r would be black almost everywhere and blinding near a pole. The log compresses that range; the tanh bounds it to [0, 1]. 2Color by where it came from, or where it's going

The very first choice is which complex number to color by. Each particle knows two: its input z and its output f(z). The Color by switch picks between them — one line at the top of calcColor:

shaders/index.ts : 333

vec2 w = (uColorBy == 0) ? z : f;   // 0 = domain (input), 1 = range (output)

Domain coloring paints each point by where it started, so the colors are a fixed map of the input plane that you watch get dragged around — the paint is glued to the plane. Range coloring paints by where it landed, so two inputs that f sends to the same place get the same color — which makes the function's many-to-one structure pop out (every sheet of a multivalued function, or every period of ez, repeats the same colors). 3Re-aiming the hue: the quantity channels

By default the hue tracks phase and the brightness tracks magnitude, but each channel can be re-pointed at a different scalar of w. The Quantity selector chooses what drives the hue:

shaders/index.ts : 339–343

if(uColorQty==1)      param = fract(0.5*log(r+1e-6));  // modulus (log, cyclic)
else if(uColorQty==2) param = 0.5 + 0.5*tanh(w.x);     // real part
else if(uColorQty==3) param = 0.5 + 0.5*tanh(w.y);     // imaginary part
else                   param = angle/TAU + 1.0;         // phase (default)

Color by modulus and you get concentric bands of hue at constant |w| (a contour map of magnitude); color by the real or imaginary part and the hue tracks one axis of the output, squashed into range by tanh. A second, independent Brightness selector (lines 348–352) does the same for the value channel — including a flat Uniform setting that turns brightness off entirely, so the hue carries all the information. Mixing the two lets you isolate exactly the feature you want to see.

In the app. Quantity, Brightness, and the contour controls below live on the viewer's Color panel, not on the embed URL. Open the full viewer and switch Quantity to Modulus while watching ez to see the magnitude contours.

4Sequential colormaps: Viridis and friends

The HSV wheel is cyclic, which is perfect for phase but misleading for a quantity that simply increases — red wraps back to red, so high and low look alike. For those, the viewer offers sequential colormaps: Grayscale, Viridis, Magma, Inferno, Plasma, Fire, Ocean. They are evaluated not from a lookup texture but from compact polynomial fits — the well-known sixth-order approximations to the matplotlib maps — so they cost a few multiplies per pixel and no memory:

src/lib/colormaps.ts : PALETTE_GLSL, the Viridis fit

vec3 cmViridis(float x){
  const vec3 c0=vec3(0.2777, 0.0054, 0.3341);
  const vec3 c1=vec3(0.1051, 1.4046, 1.3846);
  ... // six more coefficients
  return c0+x*(c1+x*(c2+x*(c3+x*(c4+x*(c5+x*c6)))));  // Horner form
}

When a colormap is active, the chosen Quantity drives its axis (magnitude is log-scaled), and the Brightness channel still modulates it. Below, ez's magnitude is shown through Viridis — the perceptually uniform ramp makes the exponential growth read as an even climb from dark blue to yellow:

5Contour bands: turning color into level sets

One more control turns the smooth coloring into a contour map. With Repeat on, the colormap axis is tiled by a mirrored, seamless triangle wave — so instead of one ramp across the whole range you get a stack of repeating bands, the complex-analysis equivalent of contour lines on a topographic map:

shaders/index.ts : 360–365

if(uColorRepeat > 0.0){
  float raw = ... lg;                              // the chosen quantity (e.g. log|w|)
  s = abs(fract(raw * uColorRepeat) * 2.0 - 1.0);  // mirrored → seamless bands
}

The abs(fract(x)*2 - 1) is the trick: fract alone would jump from 1 back to 0 at each band edge (a visible seam), but folding it into a triangle wave makes consecutive bands mirror each other, so the color is continuous across every boundary. Point it at log-magnitude and the bands are level sets of |f|; point it at phase and they are equal-angle wedges.

6Reading a picture back into math

Put it together and the default coloring is a dense readout of the function. The same field guide from the functions page applies, now grounded in the code above:

In the pictureIn the math
Dark point, full hue wheel counterclockwisea zero (r→0 dims it; phase circles once per order)
Bright point, hue wheel clockwise (reversed)a pole (r→∞; phase winds the other way)
Repeated color tiles under range coloringa many-to-one map (periods, sheets)
Banding with Repeat onlevel sets of the chosen quantity

The whole system is two independent channels — a hue and a brightness, each aimable at phase, magnitude, or a part of w, optionally through a sequential colormap and optionally banded — all riding on the single choice of coloring by the input or the output. That is enough vocabulary to make almost any complex function legible at a glance.
