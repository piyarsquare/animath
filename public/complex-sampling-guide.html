Sampling the plane

Where Complex Particles places its points before f ever acts — and why the layout changes what you see

The plane is continuous; a GPU draws a finite number of points. So before any function runs, the viewer must choose where to place its cloud of samples across the domain. That choice is quiet but consequential: the same function can look like a solid surface, a spray of fibers, or a spider's web depending only on how the input points were laid down. This guide is the sampling layer — the patterns in src/lib/particles/createParticleGeometry.ts — and how to drive it. (For the complementary question of moving samples toward where the function stretches, see the rendering guide, §5.)

The applets are live and use the pattern= parameter; the patterns read most clearly as Points (used below), and they shape the Net mode's fibers too. Drag to orbit.

1The grid — the default

The default pattern is the obvious one: a square grid. Lay √count points along each side of the domain box, each at the center of its cell. It is the right choice when you want a surface, because the regular mesh connects cleanly into the sheet and tile modes.

src/lib/particles/createParticleGeometry.ts : 144–152

const side = Math.max(1, Math.ceil(Math.sqrt(count)));
for (let i = 0; i < count; i++) {
  const ix = i % side, iy = Math.floor(i / side);
  set(i, xMin + ((ix + 0.5) / side) * spanX,    // evenly across x
        yMin + ((iy + 0.5) / side) * spanY);    // evenly across y
}

Each call to set writes the point's position and a random seed (more on that in §4). The grid treats every part of the box equally — which is exactly why it is a poor way to see anything organized around the origin, where most interesting complex behavior lives. For that, go polar.

2The radial family — rings, spokes, webs

A complex function's structure is usually radial: zeros and poles sit at points, phase circles around them, magnitude grows outward. Sampling along circles and rays rather than a grid lines the points up with that structure. The viewer offers a small family — Rings (concentric circles), Spokes (rays from the center), Web (half rings, half spokes), Polar (an interleaved lattice), and Squares (concentric square outlines). Spokes is the clearest to read in code:

src/lib/particles/createParticleGeometry.ts : 93–103 (Spokes)

const nS = Math.max(6, Math.round(Math.sqrt(count) / 2));  // number of rays
const per = Math.ceil(count / nS);                        // points per ray
for (let i = 0; i < count; i++) {
  const spoke = Math.floor(i / per), j = i % per;
  const ang = 2 * Math.PI * (spoke / nS);                 // this ray's angle
  const r = R * ((j + 0.5) / per);                        // march outward
  set(i, cx + r * Math.cos(ang), cy + r * Math.sin(ang));
}

These layouts are the domain's fibers — the level sets |z| = const (rings) and arg z = const (spokes) — so drawing them and letting f carry them is a direct picture of how the function bends circles and rays. 3Why equal-area radius matters

There is one subtlety that separates a good radial sampler from a naive one. If you space the rings evenly in radius, the points crowd near the center and thin out toward the rim, because a thin annulus at large r covers far more area than one near the origin. To keep the density uniform, the radius must grow as the square root of the ring index:

r = R·√((ring + ½) / nR)

src/lib/particles/createParticleGeometry.ts : 75 (Polar, the equal-area radius)

const r = R * Math.sqrt((ring + 0.5) / nR);   // √ → equal area per ring

The √ is exactly the inverse of "area grows like r²": annulus area is proportional to r·dr, so equal-area rings have r² evenly spaced, i.e. r spaced like √. The same idea is why a uniform sample of a disk uses r = R√u for uniform u. Without it, a radial pattern wastes most of its points in a bright knot at the center; with it, the plane is sampled evenly all the way out.

4The hidden fourth coordinate: the seed

Every sample, in every pattern, also gets a random four-component seed written alongside its position:

src/lib/particles/createParticleGeometry.ts : 62–65

const set = (i, x, y) => {
  pos[3*i] = x; pos[3*i + 1] = 0; pos[3*i + 2] = y;
  for (let k = 0; k < 4; k++) seeds[4*i + k] = Math.random();  // a vec4 per point
};

That seed is what gives each particle its own small personality on the GPU. The shader uses it for jitter (perturbing the sample so a hard grid softens into an organic cloud — either scattering the input z or fuzzing the 4-D position, per the jitter mode) and for shimmer (a per-particle phase on the time-varying brightness, so the cloud twinkles instead of pulsing in unison). The position says where; the seed says how this one wiggles.

5Which pattern, when

PatternGood for
Gridsurfaces — Sheet and Tiles need the regular mesh
Rings / Spokes / Webseeing what f does to circles and rays (the fibers)
Polaran even radial cloud that still respects the origin
Squaresaxis-aligned structure; how f bends straight contours
Randoma texture with no sampling artifacts — pure density

Sampling is the first decision in the pipeline and the easiest to overlook: it fixes which points exist before the function moves them, the projection flattens them, and the render mode draws them. Match the pattern to the structure you want to see — a grid for a surface, fibers for a flow — and the rest of the viewer has better raw material to work with.
