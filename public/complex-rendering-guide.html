How the surface is drawn

The render modes of Complex Particles — how a cloud of sampled points becomes points, a sheet, a fabric of tiles, or a net

The first two guides covered where each point goes: what the functions do to the plane, and how the 4-D graph is flattened onto your screen. This one is about the last step — what gets drawn at those positions. The same surface can be shown four ways, and the choice changes what you notice. Every mode starts from one shared placement function, surfacePos (covered in guide 2, §2), which carries a domain point through f, into 4-D, and down to a 3-D position. The render mode only decides what primitive connects those positions.

Code is quoted from src/lib/particles/createParticleGeometry.ts (the sampling) and the per-mode vertex/fragment shaders in src/animations/ComplexParticles/shaders/index.ts. The applets are the live viewer with render= set; drag to orbit.

1Points — one particle per sample

The base mode draws one particle for each domain point. So the first question is where the domain points are — and that is set by a sampling pattern. The default is a square grid: lay √count points per side across the domain box, each at a cell center.

src/lib/particles/createParticleGeometry.ts : 144–152 (the Grid pattern)

const side = Math.max(1, Math.ceil(Math.sqrt(count)));
for (let i = 0; i < count; i++) {
  const ix = i % side, iy = Math.floor(i / side);
  set(i, xMin + ((ix + 0.5) / side) * spanX,     // x across the box
        yMin + ((iy + 0.5) / side) * spanY);     // y across the box
}

The same routine offers polar layouts too — Rings, Spokes, a Web, concentric Squares — each just a different rule for placing count points; the radial ones use r = R·√((ring+½)/nR) so equal-area rings stay evenly dense rather than crowding the center. Every sampled point also gets a random 4-component seed, which the shader uses for jitter and shimmer. On the GPU each point becomes a gl_Point whose pixel size falls off with depth, and the fragment shader carves it into a disc, triangle, or pyramid:

shaders/index.ts : 480 (point-cloud vertex main, abridged) · 494–497 (disc shape)

// vertex: place the sample on the surface, size the sprite by depth
vec3 pos3 = mix(Pold, Pnew, uProjAlpha) * 1.5;  // the surfacePos pipeline
gl_PointSize = size * globalSize * (80. / -mv.z);
// fragment: a round particle — discard outside the disc, soften the rim
float r2 = dot(d, d);
if(r2 > 0.25) discard;
alpha *= 1.0 - smoothstep(0.2, 0.5, sqrt(r2));

2Sheet — connecting the samples into a surface

Points show where the surface is but leave the eye to infer the surface between them. Sheet mode fills it in. The grid samples become a mesh: each cell is two triangles, and every cell is painted one flat color — the average of the domain coloring at its four corners — so the surface reads as a smooth field of color rather than a scatter of dots.

shaders/index.ts : 593–597 (sheet-fill vertex: the four-corner average)

vec3 c = cornerColor(cellBase)
       + cornerColor(cellBase + vec2(uCellSize.x, 0.0))
       + cornerColor(cellBase + vec2(0.0, uCellSize.y))
       + cornerColor(cellBase + uCellSize);
vColor = c * 0.25;                 // one flat color per cell

A flat translucent surface would be hard to read in 3-D, so the fragment shader recovers the surface normal on the fly from how the view-space position changes across neighboring pixels — the screen-space derivatives dFdx/dFdy — and shades by how squarely that normal faces the camera. That is what makes an otherwise-flat sheet look solid:

shaders/index.ts : 641–643 (sheet fragment: normal from derivatives)

vec3 n = normalize(cross(dFdx(vViewPos), dFdy(vViewPos)));  // face normal, no mesh data
float facing = abs(n.z);                       // 1 face-on, 0 edge-on
float shade = mix(1.0, 0.45 + 0.55*facing, clamp(uShade, 0.0, 1.0));

The same mesh can be drawn as a brighter wireframe overlay (just the row and column edges, no triangle diagonals) so the grid lines stay legible over the fill. An optional directional light adds a cool, dim tint to back faces, so the inside of a folded surface is visibly distinct from the outside:

3Tiles — a fabric that tears where it stretches

Tiles sits between Points and Sheet, and it is the most revealing of the four. At every sample it places one little square, then stretches and shears that square to follow the deformed grid — the two local grid directions found by central differences of surfacePos over one cell. The trick is a per-edge length cap, uMaxTile:

shaders/index.ts : 675–682 (tile vertex: deformed grid directions, capped)

vec3 du = 0.5 * (surfacePos(z + dx) - surfacePos(z - dx));  // local u direction
vec3 dv = 0.5 * (surfacePos(z + dy) - surfacePos(z - dy));  // local v direction
vec3 duC = du * min(1.0, uMaxTile / length(du));   // clamp edge length
vec3 dvC = dv * min(1.0, uMaxTile / length(dv));
vec3 pos3 = c0 + corner.x * duC + corner.y * dvC;  // the oriented quad

Where the function is gentle, neighboring tiles are about a cell apart, fall below the cap, and meet edge-to-edge into a solid fabric. Where the function stretches the plane hard — near a pole, or out where ez blows up — the true tile spacing exceeds the cap, the clamp holds each tile to uMaxTile, and the fabric tears into a field of separated squares. So the picture shows you the local stretching directly: solid where the map is tame, scattered where it is violent. (A guard falls back to a screen-facing normal where the projection singularity would otherwise make the two edge vectors collinear and produce a NaN.)

4Net — the polar fibers of the domain

Net trades the grid for a polar lattice and draws it as threads: concentric circles and radial rays from the domain, carried onto the surface by f. It answers a different question — not "what does the surface look like" but "what does the function do to circles and rays." Each thread is a screen-space ribbon of constant pixel width: the vertex shader places both endpoints on the surface, finds the segment's direction on screen, and offsets each side perpendicular to it so the line keeps a steady thickness no matter how far away it is.

shaders/index.ts : 737–741 (fiber-net vertex: constant-width ribbon)

vec2 dirPx = (ndcB - ndcA) * uResolution;          // segment direction in pixels
vec2 dir = normalize(dirPx);
vec2 nrm = vec2(-dir.y, dir.x);                    // perpendicular
vec2 offsetNdc = nrm * aSide * (uLineWidth * 0.5) / (0.5 * uResolution);
clipA.xy += offsetNdc * clipA.w;                   // widen to constant px

Because the lattice is polar, the net is exactly the image of the domain's fibers — the circles |z| = const and rays arg z = const. Watch where 1/z sends them: circles stay circles, rays stay rays, but inside and outside trade places.

5Adaptive density — samples that follow the function

A uniform grid spends the same number of points on a flat region as on a violently stretched one — so the interesting places look sparse. The viewer fixes this two ways, one on each side of the pipeline.

On the GPU: points fill in where the sheet dissolves

Both the point cloud and the sheet measure, per cell, how far f has stretched it in 3-D — the largest of its four deformed edge lengths, cellStretch:

shaders/index.ts : 450–457

float cellStretch(vec2 base, vec2 cell){
  vec3 q00 = surfacePos(base);                     // the cell's four corners,
  vec3 q10 = surfacePos(base + vec2(cell.x, 0.0)); // placed on the surface
  vec3 q01 = surfacePos(base + vec2(0.0, cell.y));
  vec3 q11 = surfacePos(base + cell);
  return max(max(length(q10-q00), length(q01-q00)),  // longest deformed edge
             max(length(q11-q01), length(q11-q10)));
}

With Adaptive on, the sheet's fill dissolves where that stretch exceeds a threshold, and the point cloud fades in exactly there — the two are complementary, so you get a solid surface where the grid is dense and a breath of points where it has been pulled thin, with no hard seam between them.

On the CPU: more samples where the function stretches the plane

The deeper fix moves the samples themselves. Before drawing, the viewer can redistribute the domain points so their density follows |f′(z)| — the local stretch factor of the map. It estimates |f′| at each candidate by finite differences (the Frobenius norm of the Jacobian, which equals |f′| for an analytic f), then samples points in proportion to that, raised to a bias exponent α:

src/lib/particles/createParticleGeometry.ts : 208–234 (abridged)

const fz   = evalFn(x, z);
const fzh  = evalFn(x + h, z);     // finite differences in x and y
const fzhz = evalFn(x, z + h);
// Frobenius norm of the Jacobian ≈ |f'(z)| for analytic f
const mag = Math.sqrt(dux*dux + dvx*dvx + duy*duy + dvy*dvy);
weights[c] = isFinite(mag) ? mag : 0;
...
if (weights[i] > cap) weights[i] = cap;     // clamp at median × clampMul
weights[i] = Math.pow(weights[i], alpha);   // α dials the bias strength

The clamp is the subtle part. A pole like 1/z has an unbounded derivative, so without a cap nearly every sample would collapse into one bright spot. Capping each weight at the median times a factor keeps the singularity well-sampled without letting it swallow the whole budget. The final points are drawn from the cumulative distribution by binary search — weighted sampling, so denser where the function works hardest.

Try it in the app. The Adaptive toggle and the density / bias controls live on the viewer's Render and Domain panels, not on the embed URL — open the full viewer and switch Adaptive on while watching a pole (1/z) or a steep function (ez) to see the samples migrate.

6Which mode, when

ModeShows bestBuilt from
Pointsthe raw samples; clouds, jitter, density; the lightest to drawone gl_Point per sample
Sheetthe surface as a solid object; folds, self-intersections, inside vs outsidea colored mesh, derivative-shaded
Tileslocal stretching — the map's distortion, made visible by tearingone capped, oriented quad per sample
Netwhat f does to circles and rays; the polar fiber structureconstant-width ribbons on a polar lattice

All four read the same surfacePos and the same domain coloring; they differ only in the primitive they connect the samples with. That is the whole design: place the points once, draw them four ways. Combine a mode with Adaptive density and the picture also tells you where the function is doing the most work — the samples, the tiles, and the sheet all crowd toward the action.
