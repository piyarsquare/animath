1Torus — onto S³, then stereographic

The next two projections throw away the overall size of (z, f) and keep only its shape — its direction in 4D. Discarding scale means rescaling every 4-vector to unit length, which lands it on the unit 3-sphere S³ (the set of 4D points at distance 1 from the origin, just as S² is the ordinary sphere in 3D). The Torus view then carries S³ into our 3D space by stereographic projection: pick a pole, draw the line from the pole through each point, and read off where that line crosses the space — the same construction that flattens a globe onto a map, one dimension up.

P(p) = (x, y, u) / ( |p| − v )   after normalizing, where |p| = √(x²+y²+u²+v²) — stereographic from the v pole.

src/lib/viewpoint.ts : 91–101

if(mode===ProjectionMode.Torus){
  // normalize (z,f) onto S^3, then stereographically project from the
  // (0,0,0,1) pole. arg(z) runs around the hole, arg(f) around the tube,
  // and |z|/|f| selects which nested donut; the overall scale is discarded.
  const d = Math.hypot(p.x,p.y,p.z,p.w) || 1e-6;
  const denom = Math.hypot(d - p.w, POLE_EPS * d) || 1e-4;
  return new THREE.Vector3(p.x/denom, p.y/denom, p.z/denom);
}

The plain stereographic denominator would be d − p.w (the point's length minus its v component): zero exactly at the pole, where the projection shoots to infinity. The code soft-floors it by combining it with a small constant in quadrature — √((d−v)² + (ε·d)²) — so a point at the pole bends to a large-but-finite radius instead of escaping. The constant POLE_EPS is shared between the TypeScript and the shader so both bend identically:

src/lib/viewpoint.ts : 19

// Soft floor (in quadrature) on the Torus stereographic denominator, so
// points near the projection pole map to a bounded radius, not infinity.
export const POLE_EPS = 0.08;

The image is a stack of nested donuts (Clifford tori) filling space: arg z walks around the hole, arg f around the tube, and the ratio |z|/|f| chooses which donut. The Reference scaffold toggle draws the faint donuts. The Möbius map (z−1)/(z+1) makes the nesting especially clean:

2Sphere — the Hopf map

The Sphere stop goes one step further than the Torus. Instead of spreading S³ out into space, it collapses it onto an ordinary 2-sphere by the Hopf map S³ → S². Read the 4D point as a pair of complex numbers z1 = x + iy = z and z2 = u + iv = f; the Hopf map is then a tidy formula in those two:

H = ( 2 Re(z1 · z̅2),  2 Im(z1 · z̅2),  |z1|² − |z2|² ) / |(z1, z2)|²

Expanding the complex products in real coordinates gives exactly the three components in the code (z̅2 is the conjugate, so its imaginary part flips sign):

src/lib/viewpoint.ts : 78–86

if(mode===ProjectionMode.Hopf){
  const x=p.x, y=p.y, u=p.z, v=p.w;
  const d=(x*x+y*y+u*u+v*v)||1e-6;
  return new THREE.Vector3(
    2*(x*u + y*v)/d,    // 2 Re(z1 * conj z2)
    2*(y*u - x*v)/d,    // 2 Im(z1 * conj z2)
    (x*x+y*y - u*u-v*v)/d   // |z1|^2 - |z2|^2
  );
}

Every particle lands on the unit sphere, and the landing spot reads as the ratio z / f(z): the latitude encodes |z| / |f| (equator where input and output have equal magnitude, one pole where the output vanishes, the other where the input does), and the longitude encodes the phase difference arg(z) − arg(f). Because only the ratio survives, an entire circle of 4D points — the ones that differ by a common phase of input and output — collapses to a single point on the sphere. That circle is the Hopf fiber, and it is exactly the circle the Torus view kept separate. The consequence is striking:

  • f = c·z (a constant ratio) collapses to one point.
  • ez wraps the sphere infinitely.

Compare them live. The linear map below collapses the whole cloud to a single speck; swap to ez and it smears across the sphere.

Why the tumble pauses here. The Hopf and Torus maps are nonlinear, so a 4D rotation applied before them warps the picture rather than turning it. The viewer detects these two modes and freezes the 4D tumble, switching the rotation panel to rigid Yaw/Pitch/Roll camera orbits instead (createAnimationLoop.ts:67 — nonlinearMode). For the ratio reading above to hold, keep the 4D orientation at identity.

3Turning 4D: the quaternion rotations

Projection answers "how do I flatten 4D?" Rotation answers "from which angle?" In 3D you rotate around an axis; in 4D you rotate in a plane. There are six coordinate planes — xy, xu, xv, yu, yv, uv — and a turn in, say, the xu plane swaps the input's real axis with the output's real axis, mixing domain and range. The clean way to express a 4D rotation is with a pair of unit quaternions, one multiplying on the left and one (conjugated) on the right:

p  ↦  a · p · b̅  — reading the 4-vector p as a quaternion, for unit quaternions a (left) and b (right).

That sandwich product is the rotation itself. In the reference code it is three quaternion multiplies (quatMul is the Hamilton product; quatConj(b) is b̅):

src/lib/viewpoint.ts : 39–43

export function quatRotate4D(p, a, b){
  const q = new THREE.Vector4(p.x,p.y,p.z,p.w);
  const r = quatMul(quatMul(a, q), quatConj(b));   // a · p · b̄
  return new THREE.Vector4(r.x,r.y,r.z,r.w);
}

Choosing a and b independently reaches every possible 4D rotation. The viewer never asks you to type quaternions — each of the six plane buttons hands back the right pair for an angle θ from quarterQuat. The pattern is worth seeing, because it encodes a real fact about which planes are "the same kind":

src/math/quat4.ts : 11–22

export function quarterQuat(plane: string, θ: number) {
  const s = Math.sin(θ / 2), c = Math.cos(θ / 2);   // half-angle
  switch (plane) {
    case 'XY': return { L: new Q(0, 0, -s, c), R: new Q(0, 0, -s, c) };   // L = R
    case 'XU': return { L: new Q(0, s, 0, c), R: new Q(0, s, 0, c) };     // L = R
    case 'XV': return { L: new Q(s, 0, 0, c), R: new Q(-s, 0, 0, c) };    // R = conj(L)
    case 'YU': return { L: new Q(-s, 0, 0, c), R: new Q(-s, 0, 0, c) };   // L = R
    case 'YV': return { L: new Q(0, s, 0, c), R: new Q(0, -s, 0, c) };    // R = conj(L)
    case 'UV': return { L: new Q(0, 0, s, c), R: new Q(0, 0, -s, c) };    // R = conj(L)
  }
}

Two things determine each pair. First, the half-angle: a unit quaternion built from sin(θ/2) and cos(θ/2) rotates by a full θ through the sandwich product, so an eighth-turn button passes θ = 45°. Second, the left/right relationship:

  • The three imaginary–imaginary planes xy, xu, yu (built from the three vector parts) use L = R.
  • The three planes that include the v axis — the quaternion's scalar part — xv, yv, uv use R = conj(L), the sign flip on the vector part.

That split is exactly the difference between an isoclinic rotation (same angle in two orthogonal planes at once, what L = R gives) and a simple rotation in one plane (what the conjugate pairing isolates). A positive θ sends the first axis toward the negative second (+A → −B) the same way in all six planes, so the ↻ / ↺ buttons mean the same thing everywhere.

What feeds the quaternions each frame

The "Motion = Quaternion" tumble you see by default is built right in the render loop. Three plane turns at different rates are composed into a left/right pair, then multiplied onto whatever fixed orientation your buttons have accumulated:

src/lib/particles/createAnimationLoop.ts : 122–128

// Compute combined quaternion rotation
const qxy = quarterQuat('XY', tCurrent * 0.5);
const qyu = quarterQuat('YU', tCurrent * 0.7);
const qxv = quarterQuat('XV', tCurrent);
const dynL = qxv.L.clone().multiply(qyu.L).multiply(qxy.L);
const dynR = qxv.R.clone().multiply(qyu.R).multiply(qxy.R);
const Lq = dynL.clone().multiply(rotLRef.current);   // × the buttons' accumulated turn
const Rq = dynR.clone().multiply(rotRRef.current);

The three incommensurate rates (0.5, 0.7, 1.0) are why the default tumble never quite repeats — it explores 4D orientations endlessly. Lq and Rq are then pushed straight into the shader uniforms uRotL/uRotR that surfacePos reads at step 6. 4Numerical honesty: the guards

Everything above is 32-bit floating point on a GPU, and complex functions have poles — places where the output runs to infinity. The viewer is honest about where it bends the truth, and it does so at exactly the spots a complex-analysis eye looks first. You have already met two; here they are together.

Output magnitude is capped. Right after f(z) is computed, any output longer than 10³ is pulled back to that radius — so the spike at a pole of 1/z or tan z has a finite, by-construction height:

src/animations/ComplexParticles/shaders/index.ts : 439 (inside surfacePos)

vec2 fc = applyComplex(zc, functionType);
if(length(fc) > 1e3) fc = normalize(fc)*1e3;   // clamp the pole's height

Divisions floor their denominators. The individual functions never divide by a bare zero — 1/z, tan, and the rest clamp the denominator to a tiny positive number first (here 10−4):

src/lib/complexMath.ts : 49–53

export function complexInv(z: THREE.Vector2): THREE.Vector2 {
  let d = z.x * z.x + z.y * z.y;
  if (d < 1e-4) d = 1e-4;                 // 1/z never divides by zero
  return new THREE.Vector2(z.x / d, -z.y / d);
}

And the Torus pole is soft-floored by POLE_EPS (§1). One more worth naming: Γ(z) is evaluated by the Lanczos approximation with the reflection formula, so its poles at 0, −1, −2, … are genuine — accurate to a few parts in 106, far below what the eye can resolve here. The positions between the projection slider's stops, finally, are not projections at all but linear cross-fades (Part 1 §2): an animation to help your eye track points, not a map with its own meaning.
