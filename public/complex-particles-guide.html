From 4D to your screen

Part 1 of 2 — the 4-D graph and the flat projections. The math and code behind the Complex Particles viewer

Complex Particles draws a complex function f : ℂ → ℂ as a cloud of particles. Its natural home is four dimensions, and almost everything the viewer does is a way of getting those 4D points onto a flat, 3D-looking screen. This guide is about that machinery: the equations that flatten 4D to 3D, and the quaternions that let you turn the 4D object before you flatten it. It is not about any particular function — it is about the transformations every function passes through.

Each section pairs a live applet you can drag and tap with the exact source lines that compute the math. File references like src/lib/viewpoint.ts:76 point at the live source — quoted snippets are trimmed for reading. Part 1 covers the 4-D graph and the two flat projections (Perspective and Drop-axis); Part 2 takes the curved ones (Torus, the Hopf sphere) and the quaternion rotations.

1A function whose graph lives in 4D

A real function y = g(x) has a 2D graph: one axis for the input, one for the output. A complex function needs two real axes for its input z = x + iy and two for its output w = f(z) = u + iv. So its honest graph — the set of all pairs (z, f(z)) — lives in four dimensions:

AxisMeaningSource
xRe z — real part of the inputposition.x, position.z
yIm z — imaginary part of the input
uRe f(z) — real part of the outputcomputed by applyComplex
vIm f(z) — imaginary part of the output

Every particle sits at the 4D point (x, y, u, v). The input coordinates (x, y) come straight from where the particle was sampled in the plane; the output coordinates (u, v) are whatever f returns there. Each complex function is a short closed-form routine — for example ez is the textbook ex(cos y + i sin y):

src/lib/complexMath.ts : 20–23  (the GLSL twin is shaders/index.ts : 68)

// e^z = e^x (cos y + i sin y)
export function complexExp(z: THREE.Vector2): THREE.Vector2 {
  const ex = Math.exp(z.x);
  return new THREE.Vector2(ex * Math.cos(z.y), ex * Math.sin(z.y));
}

A numeric index selects which one runs — the same index in the TypeScript reference (applyComplex) and in the GPU shader, kept in lockstep so the picture and the math never disagree. We can't look at 4D directly, so the rest of this guide is the projection from those four numbers down to the three on your screen. The applet below is your drop-axis explorer for all of Part 1: it starts with the input's imaginary axis projected out (Drop Y); each button projects out one of the four axes, and Rotate returns to the full tumbling 4D view. We come back to it in §4.

2The pipeline: one function, seven steps

Before we take the projections one at a time, here is the whole journey in a single function. Every render mode — points, sheet, tiles, net — routes each vertex through surfacePos in the vertex shader. Read it top to bottom and you have read the entire 4D→3D transformation:

src/animations/ComplexParticles/shaders/index.ts : 436–447

vec3 surfacePos(vec2 zc){
  zc = domainWarp(zc);                       // 1. optional log-radial warp of z
  vec2 fc = applyComplex(zc, functionType);  // 2. f(z) -> the output (u, v)
  if(length(fc) > 1e3) fc = normalize(fc)*1e3;  // 3. clamp runaway poles
  vec2 zP = chartCoord(zc, uInCoord);        // 4. chart the input  (Cartesian/polar)
  vec2 fP = chartCoord(fc, uOutCoord);       //    chart the output
  vec4 p4 = vec4(zP.x, zP.y, fP.x, fP.y);    // 5. assemble the 4D point (x, y, u, v)
  p4 = quatRotate4D(p4, uRotL, uRotR);       // 6. rotate in 4D (the quaternions)
  vec3 Po = project(p4, uProjMode);          // 7. flatten 4D -> 3D ...
  vec3 Pn = project(p4, uProjTarget);        //    ... and the slider's other stop
  return mix(Po, Pn, uProjAlpha) * 1.5;      //    cross-fade between the two, then scale
}

Steps 5–7 are the heart of it. Step 5 packs the input and output into one 4-vector. Step 6 rotates that 4-vector (Part 2). Step 7 calls project() — twice: once for the projection slider's current stop and once for its neighbor, then mix() blends them by uProjAlpha. That blend is why dragging the slider morphs smoothly: a half-way frame is a linear cross-fade between two projections, not a projection in its own right. The rest of this guide is the project() function, one mode at a time.

Note. project() exists twice in the repo, kept identical on purpose: a readable TypeScript reference in src/lib/viewpoint.ts (used for the 4D axis cross and any CPU-side math), and the GLSL twin in shaders/index.ts that actually runs per-vertex on the GPU. We quote the TypeScript — it reads like the equations — and drop to GLSL only where the GPU path adds a guard worth seeing.

3Perspective — the 4D camera divide

The default projection is the four-dimensional analog of a camera. A 3D perspective camera divides (x, y) by depth z so distant things look smaller. Here we have one extra dimension, so we keep three coordinates and divide by the fourth:

(x, y, u, v)  ↦  (x, y, u) / (3 + v)

The eye sits at v = −3; points with larger v are "farther" and shrink toward the center. In code it is a single reciprocal scale:

src/lib/viewpoint.ts : 76

if(mode===ProjectionMode.Perspective){
  return new THREE.Vector3(p.x,p.y,p.z).multiplyScalar(1/Math.max(3+p.w,0.35));
}

Two details. First, in this 4-vector the components are named (x, y, z, w) but they mean (x, y, u, v) — so p.w is the output's imaginary part v, and p.z is the output's real part u. Second, the divide is floored: Math.max(3 + p.w, 0.35). Without it, a steep function (try ez) drives v through −3, the denominator hits zero, and that vertex flies to infinity — which on a real mobile GPU rasterizes as garbage and can crash the context. The floor keeps every point finite, and it does so continuously (no sign flip as v crosses the eye plane), so a spin doesn't pop a crossing vertex from one side to the other. The GLSL twin carries the same guard and a long comment explaining why:

src/animations/ComplexParticles/shaders/index.ts : 301–310

vec3 project(vec4 p, int mode){
  // Perspective from the eye at w = -3. The denominator 3 + p.w vanishes
  // on the eye plane (p.w = -3) ... Floor the denominator to a positive
  // minimum (no sign flip): the projection stays continuous, front-facing
  // points are unchanged, and everything stays finite.
  if(mode==0){ return p.xyz / max(3.0 + p.w, 0.35); }

4Drop-axis — the orthographic slice

The bluntest way to flatten 4D is to forget one coordinate and keep the other three. No divide, no distortion — just an orthographic slice. There are four of them, one per axis:

src/lib/viewpoint.ts : 87–90

if(mode===ProjectionMode.DropX) return new THREE.Vector3(p.y,p.z,p.w);
if(mode===ProjectionMode.DropY) return new THREE.Vector3(p.x,p.z,p.w);
if(mode===ProjectionMode.DropU) return new THREE.Vector3(p.x,p.y,p.w);
if(mode===ProjectionMode.DropV) return new THREE.Vector3(p.x,p.y,p.z);

Each line simply drops one component of (x, y, u, v) and repacks the survivors as a 3D point. Drop V keeps (x, y, u) — the input plane plus the real part of the output, the classic "real surface" view. Drop Y keeps (x, u, v) — the real input against the full complex output. Because a drop is exact and never folds, it is the honest way to check the shape the other, nonlinear projections are distorting. Use the four Drop buttons on the explorer in Fig. 1 to step through all four slices of ez and compare them — spotting which structure the curved projections (Part 2) bend.
