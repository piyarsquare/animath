What the functions do

How each complex function is computed in animath — and what that computation does to the plane it transforms

A complex function f : ℂ → ℂ takes a point of the plane and moves it to another point. In the animath viewers you give that motion a color and watch it: the input plane is painted by a rule, and f drags the paint into a new picture. This guide is about the functions themselves — the short closed-form routines that compute each one, and the geometric story each routine tells. Its companion, From 4D to your screen, covers the other half: how the 4-D graph of (z, f(z)) is flattened onto your screen.

Every function below is quoted from src/lib/complexMath.ts — the readable TypeScript reference. The same formula runs per-pixel on the GPU as a GLSL twin in each viewer's shader, kept index-for-index in lockstep so the picture and the math never disagree. The live applets are the Plane Transform viewer (left pane: the colored input; right pane: where f sends it) and, for the many-valued functions, the 4-D Complex Particles viewer.

1The color is the function's fingerprint

Before any function acts, the plane is painted by domain coloring: each complex number w = r·eiθ gets a color from its two polar parts — its angle θ (the phase) picks a hue around the color wheel, and its magnitude r sets the brightness. That is the whole code, stripped to its spine:

src/animations/ComplexParticles/shaders/index.ts : 331–352 (the calcColor core)

float r = length(w);
float angle = atan(w.y, w.x);
// phase -> hue: the angle, wrapped to one trip around the color wheel
float hue = fract(angle / TAU + hueShift);
// magnitude -> brightness: dim near 0, bright far out (log-squashed)
float val = 0.5 * (1.0 + tanh(log(r + 1e-6)));

This is the key to reading everything that follows: the color encodes the value, so once f has dragged the paint around, the new picture is f. Where the hue sweeps red→green→blue counterclockwise around a point, the function has a zero there; where it sweeps the other way, a pole; how many full color cycles you count is the order. The identity f(z) = z moves nothing, so it shows you the reference coloring itself — the two panes below are identical:

2Multiplication is rotation: z², z³

One fact drives almost every transformation here. To multiply two complex numbers you multiply their magnitudes and add their angles:

(r1eiθ1)(r2eiθ2) = r1r2·ei(θ1+θ2)

So squaring a number squares its magnitude and doubles its angle. The code never mentions angles, though — it expands (x+iy)² = (x²−y²) + i(2xy) in real arithmetic, which is the same map written in Cartesian form:

src/lib/complexMath.ts : 10–12 and 55–60

// z² = (x² − y²) + i(2xy)
export function complexSquare(z) {
  return new THREE.Vector2(z.x*z.x - z.y*z.y, 2*z.x*z.y);
}
// z³ = (x³ − 3xy²) + i(3x²y − y³)
export function complexCube(z) {
  return new THREE.Vector2(z.x*z.x*z.x - 3*z.x*z.y*z.y,
                           3*z.x*z.x*z.y - z.y*z.y*z.y);
}

Doubling every angle means the plane is wrapped around itself twice: the top half-plane alone already covers every direction once, so z² paints the full color wheel twice as you circle the origin. Watch the hue cycle through twice per loop in the image pane — that doubled cycle is the visual signature of a double zero. (z³ wraps three times; the in-app zp/q function dials the winding to any rational.)

3Division turns the plane inside out: 1/z

Dividing inverts the magnitude and negates the angle: 1/(reiθ) = (1/r)e−iθ. So 1/z swaps inside and outside of the unit circle (points near 0 fly far out; far points crowd toward 0) and reflects across the real axis. In coordinates that is the conjugate over the squared magnitude:

src/lib/complexMath.ts : 49–53

// 1/z = conj(z) / |z|²  — the negated angle is the minus on the y term
export function complexInv(z) {
  let d = z.x*z.x + z.y*z.y;
  if (d < 1e-4) d = 1e-4;          // guard: never divide by zero
  return new THREE.Vector2(z.x/d, -z.y/d);
}

The guard if (d < 1e-4) d = 1e-4 is where the math meets the machine: at z = 0 the true value is infinite, so the code floors the denominator and the viewer additionally caps the output magnitude at 10³ (see the companion guide, §8). That spot — where the hue wheel runs backwards and the surface spikes — is a pole, the exact opposite of a zero. The reciprocal map below turns the colored plane inside out around it:

4Growth and periodicity: e z

The exponential splits the input into its two parts and lets each do one job: the real part x sets the magnitude, the imaginary part y sets the angle. That is Euler's formula, and it is exactly the code:

ez = ex+iy = ex(cos y + i sin y)

src/lib/complexMath.ts : 20–23

export function complexExp(z) {
  const ex = Math.exp(z.x);                       // magnitude from Re z
  return new THREE.Vector2(ex*Math.cos(z.y),      // angle from Im z
                           ex*Math.sin(z.y));
}

Two consequences fall right out. Vertical lines become circles (fixed x → fixed radius, y sweeps the angle) and horizontal lines become rays from the origin. And because cos and sin repeat every 2π, ez is periodic with period 2πi: every horizontal strip of height 2π maps onto the whole punctured plane. 5The inverse is many-valued: ln and the branch index

Because ez sends every 2π-tall strip onto the same plane, its inverse ln cannot pick a single answer: a point has infinitely many logarithms, one per strip, all sharing a magnitude but differing in angle by multiples of 2π. The viewer makes the choice explicit with a branch index that adds whole turns to the angle:

ln(z) = ln|z| + i(arg z + 2πk),   integer k = the branch

src/lib/complexMath.ts : 188–192

export function complexLnBranch(z, branch) {
  const r = Math.hypot(z.x, z.y);
  const t = Math.atan2(z.y, z.x) + branch * 2 * Math.PI;  // + 2πk
  return new THREE.Vector2(Math.log(r), t);               // (ln|z|, arg z + 2πk)
}

Stack those branches and they form a spiral ramp — the Riemann surface — where walking once around the origin lifts you to the next sheet instead of back to the start. The roots are the same idea but with finitely many sheets, because their angle is scaled rather than free: √z halves the angle (and adds π per branch), so two laps bring you home; ∛z needs three. The code adds the branch offset before scaling, and a tiny table records how many distinct sheets result:

src/lib/complexMath.ts : 180–184, 107–111, 425–431

export function complexSqrtBranch(z, branch) {
  const t = Math.atan2(z.y, z.x) + branch * Math.PI;   // √ sheets are π apart
  const sr = Math.sqrt(Math.hypot(z.x, z.y));
  return new THREE.Vector2(sr*Math.cos(t*0.5), sr*Math.sin(t*0.5));
}
// distinct sheets: √ repeats every 2, ∛ every 3, z^(p/q) every |q|
export function branchPeriod(index, q = 1) {
  switch (index) {
    case 1: case 14: return 2;                 // sqrt, √(z(z−1)(z+1))
    case 16: return 3;                         // cbrt
    case 18: return Math.max(1, Math.abs(q));  // z^(p/q)
    default: return /* ln-family */ null;      // infinitely many
  }
}

A flat plane can't show a spiral ramp without it crashing into itself, which is exactly why these functions want the 4-D viewer: lift the sheets into the extra dimensions and they separate cleanly, meeting only at the branch point at the origin. 6Everything trigonometric is e z in disguise

The trig functions are not separate machinery — they are combinations of the exponential. From Euler's formula, sin z = (eiz − e−iz) / 2i, and the code computes it by literally calling complexExp twice:

src/lib/complexMath.ts : 25–31

export function complexSin(z) {
  const iz = new THREE.Vector2(-z.y, z.x);   // i·z (rotate by 90°)
  const e1 = complexExp(iz);                 // e^{iz}
  const e2 = complexExp(new THREE.Vector2(-iz.x, -iz.y));   // e^{−iz}
  const diff = new THREE.Vector2(e1.x - e2.x, e1.y - e2.y);
  return new THREE.Vector2(diff.y*0.5, -diff.x*0.5);        // ÷ 2i
}

Everything cascades from there. cos is the same with a plus; tan = sin/cos, and sec, csc, cot are reciprocals of those, so they inherit 1/z's poles wherever the denominator vanishes. The hyperbolic family is the identical construction without the i (sinh z = (ez−e−z)/2) — which is why sin and sinh are the same surface turned a quarter-turn in the input plane. And the inverse trig functions run the chain backward into a logarithm (arcsin z = −i·ln(iz + √(1−z²))), so they are multivalued and carry the very same branch index from §5. Real sine waves up and down; complex sin also grows exponentially as you move off the real axis, because the hidden ez takes over:

7Special shapes: Joukowski, Möbius, Gamma

A few functions earn their place by the shapes they make.

The Joukowski map ½(z + 1/z) is just the average of a number and its reciprocal — two transformations you already know, added together — yet that combination famously bends circles near the unit circle into the cross-section of an airfoil, which is how it first earned its keep in early aerodynamics:

src/lib/complexMath.ts : 70–73

export function complexJoukowski(z) {
  const inv = complexInv(z);                     // reuses 1/z, guard and all
  return new THREE.Vector2(0.5*(z.x + inv.x), 0.5*(z.y + inv.y));
}

The Möbius map (z−1)/(z+1) is a ratio of two lines; like all such fractional-linear maps it sends circles and lines to circles and lines, carrying the right half-plane neatly onto the unit disk. The Gamma function Γ(z) — the factorial extended to the whole plane — has no elementary formula at all, so it is computed by the Lanczos approximation (a short weighted sum) together with the reflection formula Γ(z) = π / (sin πz · Γ(1−z)), which folds the troublesome left half-plane onto the easy right half and, in doing so, makes the genuine poles at 0, −1, −2, … appear:

src/lib/complexMath.ts : 171–177

export function complexGamma(z) {
  if (z.x >= 0.5) return complexGammaCore(z);          // Lanczos sum directly
  // reflect the left half-plane in: Γ(z) = π / (sin(πz)·Γ(1−z))
  const s = complexSin(...PI_z);
  const g = complexGammaCore(oneMinus_z);
  return complexDiv(new THREE.Vector2(Math.PI, 0), complexMul(s, g));
}

8Reading any function at a glance

Put the pieces together and the colored picture becomes legible. Four features tell you almost everything:

What you seeWhat it meansFrom
Dark point, hues meeting counterclockwisea zero; the number of color cycles is its order§2
Bright spike, hues meeting clockwise (reversed)a pole; order = cycle count§3
A color seam you can walk around onto a new sheeta branch point of a multivalued function§5
A repeating tile of the picturea period (e.g. 2πi for ez)§4

The deeper pattern is that the whole library is built from a handful of moves — multiply (add angles), divide (invert and reflect), exponentiate (split magnitude from angle), and take logarithms (open up the sheets) — composed and combined. tan is sin over cos; sinh is sin without the i; arcsin is a ln wearing a square root; Joukowski is z plus 1/z. Learn the four moves and you can predict the picture before the GPU draws it — and when you want to know exactly how the drawing is done, every routine is a dozen readable lines away in src/lib/complexMath.ts.
