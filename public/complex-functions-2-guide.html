1The inverse is many-valued: ln and the branch index

Because ez sends every 2π-tall strip onto the same plane, its inverse ln cannot pick a single answer: a point has infinitely many logarithms, one per strip, all sharing a magnitude but differing in angle by multiples of 2π. The viewer makes the choice explicit with a branch index that adds whole turns to the angle:

ln(z) = ln|z| + i(arg z + 2πk),   integer k = the branch

src/lib/complexMath.ts : 188–192

export function complexLnBranch(z, branch) {
  const r = Math.hypot(z.x, z.y);
  const t = Math.atan2(z.y, z.x) + branch * 2 * Math.PI;  // + 2πk
  return new THREE.Vector2(Math.log(r), t);               // (ln|z|, arg z + 2πk)
}

Stack those branches and they form a spiral ramp — the Riemann surface — where walking once around the origin lifts you to the next sheet instead of back to the start. The roots are the same idea but with finitely many sheets, because their angle is scaled rather than free: √z halves the angle (and adds π per branch), so two laps bring you home; ∛z needs three. The code adds the branch offset before scaling, and a tiny table records how many distinct sheets result:

src/lib/complexMath.ts : 180–184, 107–111, 425–431

export function complexSqrtBranch(z, branch) {
  const t = Math.atan2(z.y, z.x) + branch * Math.PI;   // √ sheets are π apart
  const sr = Math.sqrt(Math.hypot(z.x, z.y));
  return new THREE.Vector2(sr*Math.cos(t*0.5), sr*Math.sin(t*0.5));
}
// distinct sheets: √ repeats every 2, ∛ every 3, z^(p/q) every |q|
export function branchPeriod(index, q = 1) {
  switch (index) {
    case 1: case 14: return 2;                 // sqrt, √(z(z−1)(z+1))
    case 16: return 3;                         // cbrt
    case 18: return Math.max(1, Math.abs(q));  // z^(p/q)
    default: return /* ln-family */ null;      // infinitely many
  }
}

A flat plane can't show a spiral ramp without it crashing into itself, which is exactly why these functions want the 4-D viewer: lift the sheets into the extra dimensions and they separate cleanly, meeting only at the branch point at the origin. 2Everything trigonometric is e z in disguise

The trig functions are not separate machinery — they are combinations of the exponential. From Euler's formula, sin z = (eiz − e−iz) / 2i, and the code computes it by literally calling complexExp twice:

src/lib/complexMath.ts : 25–31

export function complexSin(z) {
  const iz = new THREE.Vector2(-z.y, z.x);   // i·z (rotate by 90°)
  const e1 = complexExp(iz);                 // e^{iz}
  const e2 = complexExp(new THREE.Vector2(-iz.x, -iz.y));   // e^{−iz}
  const diff = new THREE.Vector2(e1.x - e2.x, e1.y - e2.y);
  return new THREE.Vector2(diff.y*0.5, -diff.x*0.5);        // ÷ 2i
}

Everything cascades from there. cos is the same with a plus; tan = sin/cos, and sec, csc, cot are reciprocals of those, so they inherit 1/z's poles wherever the denominator vanishes. The hyperbolic family is the identical construction without the i (sinh z = (ez−e−z)/2) — which is why sin and sinh are the same surface turned a quarter-turn in the input plane. And the inverse trig functions run the chain backward into a logarithm (arcsin z = −i·ln(iz + √(1−z²))), so they are multivalued and carry the very same branch index from the section above. 3Special shapes: Joukowski, Möbius, Gamma

A few functions earn their place by the shapes they make.

The Joukowski map ½(z + 1/z) is just the average of a number and its reciprocal — two transformations you already know, added together — yet that combination famously bends circles near the unit circle into the cross-section of an airfoil, which is how it first earned its keep in early aerodynamics:

src/lib/complexMath.ts : 70–73

export function complexJoukowski(z) {
  const inv = complexInv(z);                     // reuses 1/z, guard and all
  return new THREE.Vector2(0.5*(z.x + inv.x), 0.5*(z.y + inv.y));
}

The Möbius map (z−1)/(z+1) is a ratio of two lines; like all such fractional-linear maps it sends circles and lines to circles and lines, carrying the right half-plane neatly onto the unit disk. The Gamma function Γ(z) — the factorial extended to the whole plane — has no elementary formula at all, so it is computed by the Lanczos approximation (a short weighted sum) together with the reflection formula Γ(z) = π / (sin πz · Γ(1−z)), which folds the troublesome left half-plane onto the easy right half and, in doing so, makes the genuine poles at 0, −1, −2, … appear:

src/lib/complexMath.ts : 171–177

export function complexGamma(z) {
  if (z.x >= 0.5) return complexGammaCore(z);          // Lanczos sum directly
  // reflect the left half-plane in: Γ(z) = π / (sin(πz)·Γ(1−z))
  const s = complexSin(...PI_z);
  const g = complexGammaCore(oneMinus_z);
  return complexDiv(new THREE.Vector2(Math.PI, 0), complexMul(s, g));
}

4Reading any function at a glance

Put the pieces together and the colored picture becomes legible. Four features tell you almost everything:

What you seeWhat it meansFrom
Dark point, hues meeting counterclockwisea zero; the number of color cycles is its orderPt 1 §2
Bright spike, hues meeting clockwise (reversed)a pole; order = cycle countPt 1 §3
A color seam you can walk around onto a new sheeta branch point of a multivalued function§1
A repeating tile of the picturea period (e.g. 2πi for ez)Pt 1 §4

The deeper pattern is that the whole library is built from a handful of moves — multiply (add angles), divide (invert and reflect), exponentiate (split magnitude from angle), and take logarithms (open up the sheets) — composed and combined. tan is sin over cos; sinh is sin without the i; arcsin is a ln wearing a square root; Joukowski is z plus 1/z. Learn the four moves and you can predict the picture before the GPU draws it — and when you want to know exactly how the drawing is done, every routine is a dozen readable lines away in src/lib/complexMath.ts.
